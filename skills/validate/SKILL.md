# Skill: Validate

> Check the consistency and completeness of a Specs folder against itself and its source codebase.

## When to Use

- After running **Analyze** or **Expand** to verify output quality
- Before using Specs as a development guide (trust but verify)
- Periodically, to detect Specs drift from the actual codebase
- Before a release, to ensure documentation accuracy

## Parameters

| Parameter | Required | Default | Description |
|-----------|----------|---------|-------------|
| `specs_path` | Yes | — | Path to the Specs folder root |
| `project_path` | No | (inferred from `_mapping.md`) | Root of the source project |
| `checks` | No | `all` | Which check categories to run: `all`, `structure`, `content`, `code-alignment`, `dependencies` |
| `lang` | No | `en` | Language for the report output (`en` or `zh`) |

## Instructions for the Agent

You are about to validate a Specs folder for consistency and completeness. Follow these steps precisely.

### Step 1: Load the Specs Folder

1. **Read the Specs root directory** listing.
2. **Read `_mapping.md`** to get the full Spec ↔ Source mapping table.
3. **Read `00_overview.md`** to understand the project context (tech stack, core modules).
4. **Read `_glossary.md`** to get the term definitions.
5. **Build an inventory**: list all `.md` files in the Specs folder recursively, categorize them:
   - Top-level files: `00_overview.md`, `01_architecture.md`, `02_data_flow.md`, `_mapping.md`, `_glossary.md`
   - Index files: all `_index.md` files
   - Spec files: all other `.md` files (module specs)

### Step 2: Structural Checks

Verify the Specs folder structure is well-formed:

1. **Required top-level files exist**:
   - `00_overview.md` — must exist
   - `01_architecture.md` — must exist
   - `02_data_flow.md` — must exist
   - `_mapping.md` — must exist
   - `_glossary.md` — must exist

2. **Every directory has `_index.md`**:
   - Traverse all subdirectories in the Specs folder
   - Each directory that contains spec files must also have an `_index.md`
   - Report any directories missing `_index.md`

3. **YAML frontmatter validity**:
   - Every spec file (not top-level, not `_index.md`) must have valid YAML frontmatter
   - Required frontmatter fields: `status`, `maps_to`
   - `status` must be one of: `skeleton`, `draft`, `reviewed`, `stable`
   - `maps_to` must be a valid relative path

4. **No orphan specs**:
   - Every spec file should be listed in the parent `_index.md` modules table
   - Every spec file should appear in `_mapping.md`

### Step 3: Content Checks

Verify the content quality and completeness:

1. **Top-level files are not skeleton**:
   - `00_overview.md` must have a filled project description, tech stack table, and architecture diagram
   - `01_architecture.md` must have architecture decisions and module responsibilities
   - `02_data_flow.md` must have at least one end-to-end data path described

2. **Skeleton specs have minimal content**:
   - Files with `status: skeleton` must have at least a `# Module Name` heading and `## One-line Summary`
   - The one-line summary must not be a template placeholder

3. **Expanded specs are complete**:
   - Files with `status: draft` or higher must have all sections filled:
     - `## One-line Summary`
     - `## Responsibilities`
     - `## Input / Output`
     - `## Key Constraints`
     - `## Relationships`
     - `## Implementation Notes`
   - No section should contain only template placeholder text

4. **`_index.md` completeness**:
   - Each `_index.md` must list all spec files in its directory in the modules table
   - Each entry must have a non-empty description

5. **Glossary coverage**:
   - Domain terms used in specs should have definitions in `_glossary.md`
   - Check spec headings and key paragraphs for technical / domain terms

### Step 4: Code Alignment Checks

Verify that Specs accurately reflect the actual codebase:

1. **Source files exist**:
   - For every `maps_to` path in spec frontmatter, verify the source file actually exists at `project_path`
   - Report any broken references (source file deleted or moved)

2. **No significant unmapped source files**:
   - List the source directories mentioned in `_mapping.md`
   - Scan those directories for source files that have no corresponding spec
   - Ignore trivial files (configs, generated files, tests, `__init__.py`, `index.ts` re-exports)
   - Report any significant source files that lack a spec

3. **Module list consistency**:
   - Compare the list of modules described in `01_architecture.md` with the actual directories in the source
   - Report any modules that exist in code but are missing from architecture description
   - Report any modules described in architecture that no longer exist in code

4. **Tech stack accuracy**:
   - Compare the tech stack listed in `00_overview.md` with what's actually in the project's dependency files
   - Report any outdated or incorrect tech stack entries

### Step 5: Dependency Checks

Verify that inter-spec references and dependencies are consistent:

1. **Dependency paths resolve**:
   - For every spec's `dependencies` list in frontmatter, verify the referenced spec files exist
   - Report broken dependency references

2. **Bidirectional relationships**:
   - If Spec A lists Spec B as a dependency, check if Spec B's Relationships section mentions Spec A
   - Report one-directional relationships that should be bidirectional

3. **Circular dependency detection**:
   - Build the dependency graph from all spec frontmatter
   - Detect and report any circular dependency chains

4. **Data flow consistency**:
   - Cross-reference `02_data_flow.md` paths with the module relationships described in individual specs
   - Report any data flow paths that reference modules not present in the specs

### Step 6: Generate Consistency Report

Compile all findings into a structured report. Categorize issues by severity:

#### Severity Levels

| Severity | Icon | Meaning |
|----------|------|---------|
| Error | ❌ | Must fix — broken reference, missing required file, invalid frontmatter |
| Warning | ⚠️ | Should fix — incomplete content, stale information, missing coverage |
| Info | ℹ️ | Nice to fix — minor inconsistencies, style suggestions |

#### Report Format

```markdown
# Specs Validation Report

**Specs Path**: {specs_path}
**Project Path**: {project_path}
**Validated At**: {YYYY-MM-DD HH:MM}
**Checks Run**: {list of check categories}

## Summary

| Severity | Count |
|----------|-------|
| ❌ Error | {N} |
| ⚠️ Warning | {N} |
| ℹ️ Info | {N} |

**Overall Status**: {PASS / FAIL}
(FAIL if any Error exists; PASS otherwise)

## Structural Issues

- ❌ Missing required file: `01_architecture.md`
- ⚠️ Directory `src/utils/` has no `_index.md`
- ...

## Content Issues

- ⚠️ `src/core/engine.md` has `status: draft` but `## Key Constraints` is empty
- ℹ️ `_glossary.md` is missing definition for term "middleware"
- ...

## Code Alignment Issues

- ❌ `maps_to: src/core/old_engine.ts` — file does not exist (was it renamed?)
- ⚠️ Source file `src/api/v2/routes.ts` has no corresponding spec
- ...

## Dependency Issues

- ⚠️ `src/core/engine.md` depends on `src/db/client.md` but that spec does not mention engine in Relationships
- ℹ️ Circular dependency detected: engine → scheduler → engine
- ...

## Recommendations

1. [Action item based on highest-severity issues]
2. ...
```

### Step 7: Final Summary

After generating the report:
1. Print the Summary section to the user
2. If there are Errors, highlight the top 3 most critical ones
3. Suggest which Skill to use to fix the issues:
   - Missing/broken specs → re-run **Analyze**
   - Incomplete content → use **Expand** on specific files
   - Stale information → use **Diff & Upgrade** to refresh

## Output Format

After completion, report:
```
✅ Validation complete: {specs_path}
   - Files checked: {N}
   - Errors: {N}
   - Warnings: {N}
   - Info: {N}
   - Overall: {PASS / FAIL}
   - Report saved to: {specs_path}/_validation_report.md
```

## Language Handling

If `lang: zh`, generate the report in Chinese:
- Section headings: `## 结构问题`, `## 内容问题`, `## 代码对齐问题`, `## 依赖问题`, `## 建议`
- Issue descriptions in Chinese
- Keep file paths, code references, and technical terms in English

If `lang: en` (default), generate the report in English.

## Tips for Quality

- **Be precise about locations**: Always reference the exact file and line/section where an issue was found.
- **Suggest fixes, not just problems**: For each issue, briefly indicate how to resolve it.
- **Prioritize actionable issues**: Errors that break the Specs' usability come first; cosmetic issues come last.
- **Don't flag intentional design choices**: If a module is intentionally unmapped (e.g., test files), don't report it as missing.
