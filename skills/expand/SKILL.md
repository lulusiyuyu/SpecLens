# Skill: Expand

> Expand a skeleton Spec into a detailed specification by deeply analyzing the corresponding source code.

## When to Use

- You have a skeleton Spec generated by the **Analyze** skill and want to fill in details
- You want to deeply understand a specific module
- You want to prepare a module's Spec for use as a development guide

## Parameters

| Parameter | Required | Default | Description |
|-----------|----------|---------|-------------|
| `spec_path` | Yes | — | Path to the skeleton Spec file to expand |
| `project_path` | No | (inferred from spec) | Root of the source project, if not inferrable |
| `lang` | No | (from spec frontmatter) | Language for output (`en` or `zh`) |

## Instructions for the Agent

You are about to expand a skeleton Spec file into a fully detailed specification. Follow these steps precisely.

### Step 1: Read the Skeleton Spec

1. **Read the spec file** at `spec_path`.
2. **Extract from YAML frontmatter**:
   - `maps_to` — the corresponding source file(s)
   - `dependencies` — related specs
   - `status` — should be `skeleton` or `draft`
   - `lang` — language preference
3. If `status` is already `reviewed` or `stable`, warn the user and ask for confirmation before overwriting.

### Step 2: Locate and Read the Source Code

1. **Find the source file** using the `maps_to` path. If it's relative, resolve it from the project root.
2. **Read the entire source file** carefully.
3. **Also read**:
   - Files that this module imports/requires (to understand dependencies)
   - Files that import/use this module (to understand who depends on it)
   - Test files for this module (if they exist, to understand expected behavior)
   - Type definition files or interface files (if separate)

### Step 3: Deep Analysis

Analyze the source code for:

1. **Responsibilities**: What does this module do? Break it into a clear bullet list.
2. **Public Interface**: What functions/classes/methods are exported? What are their signatures?
3. **Input / Output**: What does the module receive (parameters, events, data) and produce (return values, side effects, emitted events)?
4. **Key Constraints**:
   - Performance considerations (async operations, caching, batching)
   - Security considerations (input validation, sanitization, auth checks)
   - Edge cases and boundary conditions
   - Invariants that must always hold
5. **State Management**: Does it maintain internal state? How is state mutated?
6. **Relationships**:
   - Direct dependencies (what it imports)
   - Reverse dependencies (who imports it)
   - Communication patterns (function calls, events, messages, shared state)
7. **Key Algorithms / Logic**: Notable implementation details — state machines, retry logic, parsing strategies, optimization techniques.
8. **Error Handling**: How does the module handle errors? What errors can it throw?

### Step 4: Fill in the Spec

Using the template format from `templates/spec_file.md`, fill in **every section** with detailed content:

#### `## One-line Summary`
One clear sentence: what this module does and why it exists.

#### `## Responsibilities`
Bullet list of responsibilities. Each bullet should be a concrete, testable statement.

Example:
```markdown
- Parse incoming HTTP requests and route them to the correct handler
- Validate request parameters against the schema before processing
- Return standardized error responses for invalid requests
```

#### `## Input / Output`
A table describing all inputs and outputs:

| Direction | Type | Description |
|-----------|------|-------------|
| Input | `Config` object | Configuration loaded at startup |
| Input | HTTP Request | Incoming API requests |
| Output | HTTP Response | JSON response to client |
| Output | Log entries | Writes to application logger |

#### `## Key Constraints`
Specific, measurable constraints:

```markdown
- Must handle up to 1000 concurrent connections
- Response time must be < 100ms for cached queries
- Must validate all user input before database operations
```

#### `## Relationships`
A table of all module relationships:

| Relation | Target | Description |
|----------|--------|-------------|
| depends_on | `src/config/loader.ts` | Reads configuration at startup |
| depends_on | `src/db/client.ts` | Uses for database queries |
| depended_by | `src/api/routes.ts` | Routes delegate to this handler |

#### `## Implementation Notes`
Detailed technical notes:
- Key algorithms with brief explanations
- State machine descriptions (with Mermaid diagrams if applicable)
- Design patterns used
- Notable code paths
- Any technical debt or TODOs found in the code

### Step 5: Update Frontmatter

1. Change `status: skeleton` → `status: draft`
2. Set `last_expanded` to today's date (YYYY-MM-DD format)
3. Update `dependencies` if you discovered new ones during analysis
4. Ensure `spec_type` is set correctly:
   - `architecture` for structural/system-level specs
   - `feature` for feature-specific specs
   - `interface` for API/interface definitions
   - `test` for test strategy specs
   - `constraint` for constraint/requirement specs

### Step 6: Update Related Files

1. **Update `_mapping.md`**: Change the status column for this spec from `skeleton` to `draft`
2. **Update parent `_index.md`**: If the one-line description can now be improved based on deeper understanding, update it
3. **Update `_glossary.md`**: If you encountered new domain terms during analysis, add them

### Step 7: Final Review

Before finishing:
1. Re-read the expanded spec from top to bottom
2. Verify every section has meaningful content (not template placeholder)
3. Verify all technical claims are supported by the source code you read
4. Verify relationships are accurate and complete
5. Verify YAML frontmatter is valid and consistent

## Output Format

After completion, report:
```
✅ Expanded: {spec_path}
   - Status: skeleton → draft
   - Source analyzed: {maps_to}
   - Dependencies found: {N}
   - Related files updated: _mapping.md, _index.md, _glossary.md
```

## Language Handling

If `lang: zh`, use Chinese for all content:
- Section headings: `## 一句话描述`, `## 职责`, `## 输入/输出`, `## 关键约束`, `## 与其他模块的关系`, `## 实现笔记`
- Descriptions and prose in Chinese
- Keep code references, file paths, and technical identifiers in English

If `lang: en` (default), use English for all content.

## Tips for Quality

- **Be specific, not vague**: "Handles user authentication" is too vague. "Validates JWT tokens from the Authorization header, checks token expiry, and extracts user ID for downstream handlers" is specific.
- **Describe What and Why, not How**: Focus on purpose and constraints, not implementation details line-by-line. Implementation Notes is the only section for How.
- **Be honest about unknowns**: If something is unclear from the code, say so rather than guessing.
- **Think like a newcomer**: Write for someone who has never seen this code before.
